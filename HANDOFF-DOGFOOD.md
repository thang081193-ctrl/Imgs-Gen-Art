## Dogfood checklist

### 1. Refresh policy rules from real sources (first manual scrape)

```bash
fnm exec --using=20 bash -lc "npm run scrape-policy-rules"
```

This pulls fresh content from the configured Meta + Google Ads + Play
policy URLs (PLAN-v3 §4.2 sources). Validates that the C2 scraper still
runs cleanly against live HTML (carry-forward §13 — "real-world fixture
refresh").

**Capture:**
- Did all 3 platforms fetch successfully? (failures = source URL drift)
- Bi-weekly freshness banner state in the UI
- Any new `<main>` blocks / structure changes that break extraction

### 2. Fire 5 batches per lane against real profiles

Use the chartlens / ai-chatbot / plant-identifier seeded profiles
(or your own). Walk all 3 wizards Step 1 → Step 4. Mix happy paths +
edge cases:

- **Meta Ads** (`/wizard/meta-ads`) — 5 batches × varied featureFocus
  (restore, enhance, ai_art, three_d, cartoon)
- **Google Ads** (`/wizard/google-ads`) — 5 batches × varied
  (headlineCount × descriptionCount). Note: LLM provider is unwired by
  default — runner falls back to deterministic synth. To test real
  LLM output, register a provider via the LLM service layer first.
- **Play ASO** (`/wizard/play-aso`) — 5 batches × different targetLangs
  combinations (cap = 3 langs)

**Capture per batch:**
- Did `policy_blocked` ever fire? On what keyword/rule?
- Did `policy_warned` fire? Was the override flow ergonomic?
- Were any TRUE policy issues missed (false-negative)?
- Did any LEGITIMATE copy trip a block (false-positive)?
- Latency from "Run batch" click to first asset (target <5s for mock,
  varies for real LLM/image providers)

### 3. Audit blob round-trip checks

```bash
sqlite3 data/assets.db "SELECT id, status, json_extract(policy_decision_json, '$.ok') AS ok FROM batches WHERE policy_decision_json IS NOT NULL ORDER BY started_at DESC LIMIT 20;"
```

- Verify decision JSON is well-formed
- Verify Q-44.F: clean batches have NULL `policy_decision_json`
- Verify override-cleared batches DO persist the decision (override row
  is the audit signal)

## Metrics to track (week-over-week)

Recommended log file: `data/dogfood-week1.md` (gitignored or PR'd as
soak data — bro's call).

| Metric | Target | Notes |
|---|---|---|
| False-positive rate | <5% of warned/blocked runs | Expected non-zero given placeholder seeds |
| False-negative count | 0 in obvious cases | "miracle" / "unbeatable" / "lawsuit" / "click here" / "free download" / "5 stars" must trigger |
| Override frequency | <30% of warned runs | Higher = warning rules too noisy; tune severity |
| Runner-start preflight latency | <100ms | Add timing hook if surfaces as user pain |

| Wizard "Resolve overrides" round-trip | clean re-check after confirm | Confirm decision re-fires + Run unlocks |
